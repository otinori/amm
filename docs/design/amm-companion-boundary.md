# AMM / Companion 責務境界

ステータス: `amm` 側の作業メモ。本書は Companion の設計書の写しではない。
ワークスペース / レンダラ / アーティファクトモデルの設計は引き続き Companion が source of truth。

## 意図

- `Companion` はワークスペースサーバ兼アーティファクトシステム
- `amm` はマルチエージェント実行 / オーケストレーション層
- `amm` は Windows 上における `amm` の主要ランタイム / UI 資産

このメモは `amm` のスコープが膨張するのを防ぐ目的で存在する。

## 責務分担

### Companion が担う

- ワークスペースモデル
- アーティファクトの永続化
- アノテーション / 参照モデル
- レンダラホスト / レジストリ / レンダラのライフサイクル
- ホワイトボードおよびレンダリング機能
- MCP に向けたワークスペースサーバの振る舞い
- アーティファクト安全な編集オペレーションと検証

### amm が担う

- 複数 AI エージェントセッションの起動
- ペイン / セッションのライフサイクル管理
- エージェントプロセスの I/O 集約
- 入力待ち状態の検出と「叩き起こし」挙動
- ブロードキャスト / 選択的なコマンド注入
- オペレータ向けのセッション制御 UI
- 人間オペレータと多数エージェント間のランタイム協調

### amm が再実装すべきでないもの

- アーティファクトの保存モデル
- ホワイトボードの状態モデル
- レンダラ内部
- アノテーションの永続化
- アーティファクト間の参照セマンティクス

## 統合モデル

推奨形態:

- `Companion` をローカルワークスペースサーバとして起動
- `amm` をローカルのオーケストレーションクライアント / オペレータアプリとして起動
- `amm` は安定したローカル API 境界を経由して `Companion` に接続
- `amm` は エージェント / セッション操作向けに自身のローカル制御サーフェスも公開してよい

これにより、実行プレーンとワークスペースプレーンを分離して保つ。

## Companion から最低限必要な API

`amm` は Companion の内部にフルアクセスする必要はなく、オペレータ向けの小さなサーフェスがあれば足りる。

### ワークスペース / セッションコンテキスト

- `workspace.open`
  ローカルワークスペースをオープンまたはアタッチ
- `session.create`
  ワークスペース内にオペレータまたはエージェントセッションを作成
- `session.attach`
  既存のセッション / ワークスペースコンテキストに `amm` を再アタッチ
- `session.list`
  オペレータに関連するアクティブセッションの発見

### アーティファクトナビゲーション

- `artifact.open`
  URI またはワークスペースパスでアーティファクトを開く
- `artifact.resolve`
  ロケータ / 参照ターゲットを表示可能なターゲットに解決
- `artifact.list`
  オペレータルーティング用にアーティファクト候補を列挙

### 編集 / アノテーションのハンドオフ

- `edit.apply`
  `amm` 側で直接編集せずに、編集要求を Companion に submit する
- `annotation.create`
  Companion 所有のアノテーションを作成
- `reference.create`
  Companion 所有の参照を作成

### レンダラ / ホワイトボードのハンドオフ

- `renderer.launch`
  Companion に対し、適切なレンダラのオープン / アクティブ化を依頼
- `renderer.reveal`
  特定のアーティファクト / ロケータにレンダラのフォーカスを当てる
- `whiteboard.open`
  Companion が管理するボードまたはボード相当のレンダリング面を開く

## amm 側が公開し得る最低限の API

Companion または外部ツールが `amm` を制御する必要がある場合でも、サーフェスは狭く保つ。

- `agent.list`
- `agent.start`
- `agent.stop`
- `pane.list`
- `pane.focus`
- `pane.send`
- `pane.broadcast`
- `pane.capture`
- `session.status`

これらは実行 / オーケストレーションに関する関心事であり、ワークスペースの関心事ではない。

## トランスポート選定

現状の推奨:

- `amm` と `Companion` のリポジトリ間統合では、ローカル `http` を優先
- 単一プロセスや CLI 組み込みが有用な場面では `stdio` を併設

理由:

- `Companion` は既に `http` と `stdio` のスキャフォルディング済み
- `amm` は長寿命のデスクトップ / オペレータプロセスとして動く可能性が高い
- ローカル HTTP は再接続・診断・複数クライアントのアタッチに扱いやすい

## 運用ルール

- `amm` はワークスペース状態については Companion を「正」とみなして扱う
- `amm` はアーティファクトやアノテーションの状態を並行コピーで永続化しない
- 応答性向上のためのキャッシュは可。ただしキャッシュは捨ててよいもの
- 双方に属し得る操作は、明らかにセッションオーケストレーションでない限り Companion 側に既定で寄せる

## amm 側の直近実装方針

直近の `amm` 側作業は以下に集中する:

- `amm` の堅牢化
- .NET 上での エージェント / ペイン / セッション抽象の定義
- Companion 向けクリーンなローカル統合クライアントの追加
- オペレータワークフローに有用なセッションメタデータの収集
- レンダラ / ホワイトボード関連はスコープ外に置く

- `amm` のオーケストレーション API は MCP 経由で公開するのか、素の HTTP か、両方か
- `amm` 上のオペレータ操作を Companion の audit / 履歴へどう反映させるか
